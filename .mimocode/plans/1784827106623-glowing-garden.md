# Implementation Plan: Discord Message Deletion Synchronization

## Objective
When a Discord message is deleted from a monitored channel, automatically remove the corresponding issue record from the `issues` table.

## Changes Required

### 1. `src/discord_mcp/issues_repository.py` — Add delete method

Add a new `delete_issue_by_message_id` method to `IssuesRepository`:

```python
async def delete_issue_by_message_id(self, discord_message_id: str) -> bool:
```

- Executes `DELETE FROM issues WHERE discord_message_id = %(discord_message_id)s`
- Returns `True` if a row was deleted, `False` if no matching record exists
- Uses the same `self._pool.connection()` + `conn.transaction()` pattern as `insert_issue`
- Parameterized query to prevent SQL injection

### 2. `src/discord_mcp/issue_capture.py` — Add deletion service method

Add a `delete_message` method to `IssueCaptureService`:

```python
async def delete_message(self, message_id: str) -> IssueCaptureResult:
```

- If repository is disabled (`self._repository is None`), return `DISABLED` status
- Call `self._repository.delete_issue_by_message_id(message_id)`
- Log: deletion detected, issue deleted, no matching issue found
- Catch database exceptions and return `DATABASE_FAILED` status
- Reuse existing `IssueCaptureStatus` enum values

Also update `IssueCaptureStatus` enum — no new values needed, existing ones suffice:
- `DISABLED` when no database
- `DATABASE_FAILED` on DB error
- A new status `DELETED` for successful deletion (add to enum)

### 3. `src/discord_mcp/server.py` — Add `on_message_delete` event handler

Add a new `@bot.event` handler after the existing `on_message`:

```python
@bot.event
async def on_message_delete(message):
```

Logic:
1. `await bot_ready_event.wait()` — same readiness gate as `on_message`
2. Check `monitored_channels_config.enabled` — if no channels configured, return early
3. Check `message.guild is not None` — ignore DMs
4. Check `monitored_channels_config.matches_channel(message.channel)` — ignore unmonitored channels
5. Call `await issue_capture_service.delete_message(str(message.id))`
6. Wrap in try/except, log any unhandled exceptions (do not propagate)

### 4. `tests/test_issue_capture.py` — Add deletion tests

Update `FakeRepository` to support deletion:

```python
async def delete_issue_by_message_id(self, discord_message_id: str) -> bool:
```

Add test cases to `IssueCaptureServiceTests`:

| Test | Scenario |
|------|----------|
| `test_delete_removes_matching_issue` | `delete_issue_by_message_id` returns `True` → status is `DELETED` |
| `test_delete_no_matching_issue` | returns `False` → status is `DELETED` (still success, no error) |
| `test_delete_database_error` | raises `RuntimeError` → status is `DATABASE_FAILED` |
| `test_delete_disabled_capture` | repository is `None` → status is `DISABLED` |

### 5. `tests/test_issues_repository.py` — No changes needed

The repository tests currently only test serialization. The delete method follows the same pattern as `insert_issue` and would require a live database to test. This is consistent with how `insert_issue` is tested (it isn't unit-tested in isolation).
