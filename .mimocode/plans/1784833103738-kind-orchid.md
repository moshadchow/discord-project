## Step 1: Add write methods to repository

**File:** `src/discord_mcp/api/repository.py`

Rename class from `IssueReadRepository` to `IssueRepository` (now supports reads + writes).

Add two methods:

### `create_issue(row: IssueRow) -> IssueRow`
- Insert a new row into the `issues` table
- Use SQLModel's `session.add()` + `session.commit()`
- Return the inserted row (with auto-generated `id`, `created_at`, `updated_at`)

### `delete_by_discord_message_id(discord_message_id: str) -> bool`
- Delete the row matching the `discord_message_id`
- Return `True` if a row was deleted, `False` if no match
- Use SQLModel's `session.exec(delete(...))` or `session.delete()`

## Step 2: Update service layer

**File:** `src/discord_mcp/api/service.py`

### Update `reply_to_issue()` (currently in routes, move to service)

```python
async def reply_to_issue(
    self, discord_message_id: str, message: str, client
) -> Optional[dict]:
```

Flow:
1. Look up original issue via `get_by_discord_message_id()`
2. If not found, return `None`
3. Fetch channel, send message via Discord client
4. Capture Discord response (new message ID, timestamp)
5. Build new `IssueRow` with:
   - `discord_message_id` = new message ID from Discord
   - `guild_id`, `guild_name`, `channel_id`, `channel_name` = from original issue
   - `sender` = configurable (default `"BOT"`)
   - `issue_date` = today's date
   - `issue_time` = current time
   - `issue` = the reply message text
   - `attachments` = `"[]"` (empty JSON array)
   - `message_timestamp` = Discord response timestamp
   - `message_timestamp_local` = converted local timestamp
6. Insert via `repository.create_issue()`
7. Return success with new record data

### Update `delete_issue_message()`

Current: deletes Discord message, returns success.
Spec: delete Discord message → then delete DB record → return success with both flags.

```python
async def delete_issue_message(
    self, discord_message_id: str, client
) -> Optional[dict]:
```

Flow:
1. Look up issue via `get_by_discord_message_id()`
2. If not found, return `None`
3. Delete Discord message (try/except, raise on failure)
4. Delete DB record via `repository.delete_by_discord_message_id()`
5. If DB delete fails, log error (Discord already deleted, manual reconciliation needed)
6. Return `{success, discord_message_id, channel_id, message_deleted, database_record_deleted}`

## Key Constraints

- Reply record must be a **new insert**, never update the original issue
- DB delete must happen **only after** successful Discord deletion
- If Discord fails, DB record must remain unchanged
- If DB fails after Discord success, log for manual reconciliation
- Reuse existing Discord client via `get_discord_client` dependency
- Configurable sender (default `"BOT"`, passed as parameter or env var)
